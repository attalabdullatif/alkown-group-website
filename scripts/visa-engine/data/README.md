# Visa rules — draft datasets for staff verification

Draft visa-rule records in the import-template shape (see
`../templates/visa_rules.template.json`). Every row here is a **draft**:
`confidence_level: LOW`, `status: REQUIRES_MANUAL_REVIEW`. Nothing in this
folder is "truth" until a staff member confirms it against the official
portal in `official_website` and flips it to `VERIFIED` (via the Visa Admin
page or a re-import with `status: VERIFIED`).

## Sourcing policy

- **Primary sources only for VERIFIED data**: government MFA / immigration /
  eVisa portals, embassies, border authorities.
- **`evisaguides.com` is NOT a source.** Its `robots.txt` sets
  `ai-train=no` and `Disallow: /` for ClaudeBot/GPTBot/CCBot, i.e. it
  prohibits automated/AI collection. Do not scrape it. It may only be used
  as a human-browsable cross-reference during manual verification.
- Drafts compiled from secondary aggregators (Wikipedia/Henley/etc.) carry
  `source_type: secondary` + `confidence_level: LOW` and MUST be verified
  before publishing.

## Files

- `syria-batch1.json` — Syrian passport (SY) → 26 destinations (Gulf/Arab
  neighbours, popular eVisa/VOA, and the main embassy destinations TR/DE/FR/
  GB/US/CA/JP). Includes a correction to the legacy `SY_AE` rule (UAE is
  **visa-required** for Syrians, not visa-on-arrival).
- `syria-batch2.json` — Syrian passport (SY) → 28 more destinations
  (remaining Gulf OM/KW, transit neighbours GE/AZ/RU/AM/RS/BY, Asia
  SG/PH/LK/BD/HK/MO/KG, African eVisa ET/KE/TZ/UG/ZW/ZM/NG/GH, Americas
  CU/SR/DM/CO, plus MA).

- `syria-inbound.json` — INBOUND to Syria (nationality X → destination SY),
  136 rows, generated by `../harvest/syria-inbound.js` from the official
  Syrian entry-visa fee schedule (country-group table). `government_fee` =
  the base tier (single entry, 1 month); the full 4-tier fee table (single/
  double/multiple/transit) is in `notes_ar`. Group 1 (LB, JO, MY, MR) is
  visa-free. `source_type: government`. ⚠️ Group membership was transcribed
  from the source image — verify each country's group before publishing,
  since the fee depends on it. Regenerate after edits: `node
  scripts/visa-engine/harvest/syria-inbound.js`.

Both Syria batches: basis = "Visa requirements for Syrian citizens"
(Wikipedia/Henley, mid-2026) + known official portals. Verify each row at its
`official_website` before marking VERIFIED. Syrian policy changed heavily
after 2024 — re-confirm, do not trust blindly. Where 2026 aggregators
conflicted (e.g. Gulf states), rows were marked conservatively as
`visa_required` pending verification.

## `generated/` — broad coverage from Wikipedia

`generated/<code>.json` is produced by
`../harvest/from-wikipedia.js`, which converts the "Visa requirements for
X citizens" tables (Wikipedia, CC BY-SA — reuse allowed with attribution)
into draft rows. One file per nationality, ~193 destinations each, for all
22 Arab passports. Regenerate any time:

```bash
node scripts/visa-engine/harvest/from-wikipedia.js PS    # one nationality
node scripts/visa-engine/harvest/from-wikipedia.js all   # all 22 (polite delays)
```

These are **breadth, not depth**: verdict + allowed stay + Wikipedia notes
only (no fees, no official portal, no Arabic). They exist so every
nationality has a complete matrix to verify against; staff enrich + verify
each before VERIFIED.

**Syria precedence:** import `generated/sy.json` first for breadth, then
`syria-batch1.json` + `syria-batch2.json` to overwrite those 54 routes with
the higher-quality curated data (official portals, fees, Arabic notes, the
SY→AE correction). The importer upserts by route key, so last-imported wins.

## Enriching `official_website`

Destination-level official portals (verified 2026-04-01) live in
`../harvest/destination-sources.js`. Apply them to draft rows:

```bash
node scripts/visa-engine/harvest/enrich-sources.js --all --dry-run  # report coverage
node scripts/visa-engine/harvest/enrich-sources.js --all            # fill empty official_website
```

Only fills EMPTY fields (curated portals are never overwritten) and never
changes the verdict/confidence/status. Currently covers ~87 eVisa/ETA
destinations (≈1,950 rows): the staff-verified 2026-04-01 set plus a
`WEB_RESEARCHED` block (NG, GH, AE, MV, BD, ZA, UA, LB, BW, NP, GM) added
2026-06-28 from official government domains. The dry-run lists destinations still without a
portal — mostly embassy-visa destinations (Schengen states, etc.) that have
no single eVisa portal, plus any not yet supplied. Add new ones to
`destination-sources.js` and re-run. Special cases handled manually:
Iraqi Kurdistan (`visit.gov.krd`) and Schengen/ETIAS
(`travel-europe.europa.eu`, visa-exempt travellers only — not for
visa-required nationalities).

## Import

```bash
# from repo root, with REACT_APP_SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY set

# One command — imports every dataset in the correct order
# (generated/* → syria-batch1/2 → syria-inbound):
node scripts/visa-engine/import-all.js --dry-run   # validate everything, no writes
node scripts/visa-engine/import-all.js             # upsert everything

# Or a single file:
node scripts/visa-engine/import-json.js scripts/visa-engine/data/<file>.json --dry-run
node scripts/visa-engine/import-json.js scripts/visa-engine/data/<file>.json
```

Rows land as `REQUIRES_MANUAL_REVIEW`, so the public site is unaffected until
staff verify them.
